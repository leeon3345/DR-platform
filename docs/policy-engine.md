# Policy Engine 설명

이 문서는 DR Platform에서 복구 정책이 어떻게 저장되고, 어떤 데이터를 기준으로 복구 추천 점수가 계산되며, 그 결과가 UI에 어떻게 표시되는지 설명합니다.

## 한 줄 요약

Policy Engine은 네임스페이스별 복구 정책과 현재 클러스터 상태를 비교해 “어떤 워크로드를 먼저 복구해야 하는지”를 점수와 설명으로 보여주는 로직입니다.

## 입력 데이터

| 입력 | 수집 위치 | 사용 목적 |
| --- | --- | --- |
| 복구 정책 | `drctl policy set`, `/api/clusters/:id/recovery-policy` | 네임스페이스별 중요도, RTO, RPO 기준 정의 |
| 워크로드 상태 | `dr-agent`, Kubernetes Pod 목록 | 장애 여부와 Pod 정상 동작 여부 판단 |
| 백업 이력 | `dr-agent`, Velero backup 목록 | 최신 복구 지점이 충분히 최신인지 판단 |
| 장애 이벤트 | Alertmanager webhook | UI 경고 표시와 추천 표시 트리거 |
| 승인 상태 | `/recommendations/:workloadId/approve` | 운영자가 승인한 복구 대상 추적 |

## 정책 등록 방식

운영자는 CLI로 네임스페이스별 정책을 등록합니다.

```bash
drctl policy set my-cluster \
  --namespace order-service \
  --tier critical \
  --rto 1h \
  --rpo 30m
```

정책 항목의 의미는 다음과 같습니다.

| 필드 | 의미 |
| --- | --- |
| `namespace` | 복구 정책을 적용할 Kubernetes namespace |
| `tier` | 서비스 중요도. `critical`, `high`, `normal`, `low` 중 하나 |
| `rto` | 장애 감지 후 목표 복구 시간 |
| `rpo` | 허용 가능한 데이터 손실 기준, 즉 백업 최신성 목표 |

## 점수 계산 방식

현재 구현의 추천 점수는 세 가지 항목의 가중합입니다.

```text
복구 추천 점수 =
  Tier 중요도 점수 * 40%
+ 백업 최신성 점수 * 40%
+ 워크로드 상태 점수 * 20%
```

### 1. Tier 중요도 점수

| tier | 점수 |
| --- | ---: |
| `critical` | 100 |
| `high` | 70 |
| `normal` | 40 |
| `low` | 10 |

중요한 서비스일수록 복구 우선순위가 높아지도록 반영합니다.

### 2. 백업 최신성 점수

| 최신 성공 백업 나이 | 점수 |
| --- | ---: |
| 백업 없음 | 20 |
| 60분 이내 | 100 |
| 6시간 이내 | 80 |
| 24시간 이내 | 50 |
| 24시간 초과 | 20 |

최신 백업이 있을수록 복구 가능성이 높다고 보고 높은 점수를 부여합니다. 반대로 오래된 백업만 있거나 백업이 없으면 복구 리스크가 높다고 판단합니다.

### 3. 워크로드 상태 점수

| Pod 상태 | 점수 |
| --- | ---: |
| Failed Pod 있음 | 0 |
| Pending Pod 있음 | 50 |
| 모든 Pod Running | 100 |
| 그 외 불완전 상태 | 50 |

장애가 실제 Pod 상태로 나타나면 점수에 즉시 반영됩니다.

## 추천 순위 정렬

점수 계산이 끝나면 다음 기준으로 복구 추천 순위를 정합니다.

1. 추천 점수가 높은 순서
2. 점수가 같으면 `tier` 중요도가 높은 순서
3. 그래도 같으면 namespace 이름순

UI의 `AI Recovery Decision` 테이블은 이 순서를 그대로 보여줍니다.

## 장애 탐지와 UI 경고

Alertmanager에서 장애 이벤트가 들어오면 서버는 이벤트를 저장하고, 대시보드는 이를 기반으로 경고 상태를 표시합니다.

- 네임스페이스 단위 alert가 있으면 해당 namespace가 `warning` 또는 `outage` 상태로 표시됩니다.
- 클러스터 단위 alert가 있으면 클러스터 topology가 경고 또는 장애 상태로 표시됩니다.
- 장애 이벤트가 없으면 추천 영역은 “장애 이벤트 없음. 모니터링 중...” 상태를 보여줍니다.

중요한 점은 Alertmanager의 `severity`가 현재 점수 산식에 직접 더해지지는 않는다는 것입니다. 장애 이벤트는 UI 상태와 추천 표시 트리거에 사용되고, 점수에는 장애로 인해 변화한 Pod 상태가 반영됩니다.

## RTO와 RPO 반영 방식

`RTO`는 복구 승인 이후 실제 복구 완료까지의 시간을 기록하고, 정책 목표와 비교하기 위한 기준으로 사용됩니다.

`RPO`는 정책 값으로 저장되고 추천 설명에 포함됩니다. 현재 백업 최신성 점수는 RPO 값 자체를 직접 파싱해 비교하기보다는 60분, 6시간, 24시간 기준의 고정 구간으로 계산합니다.

따라서 설명할 때는 다음처럼 말하는 것이 정확합니다.

```text
RPO는 정책으로 저장되어 UI와 추천 설명에 포함되며,
백업 최신성 평가는 현재 구현상 고정 시간 구간 기준으로 점수화된다.
```

## 추천 설명 생성

각 추천 항목에는 사람이 읽을 수 있는 설명이 붙습니다.

설명에는 다음 정보가 포함됩니다.

- namespace
- tier
- 추천 점수
- 최신 백업 나이
- Pod 상태 요약
- RTO/RPO 정책 값
- 운영자 승인 여부

LLM API를 사용할 수 있으면 장애가 있는 상위 추천 항목에 대해 AI 설명을 생성합니다. 실패하거나 키가 없으면 서버가 기본 설명 문구를 생성합니다.

## 복구 승인 흐름

운영자가 UI에서 추천 항목을 승인하면 `/api/clusters/:id/recommendations/:workloadId/approve`가 호출됩니다.

승인 이후 서버는 해당 workload의 복구 요청 상태를 저장하고, agent command queue 또는 복구 orchestration 흐름으로 연결합니다. UI는 승인됨, 복구 대기, 실행 중, 실패, 완료 상태를 표시합니다.

## 구현 위치

| 파일 | 관련 로직 |
| --- | --- |
| `server/server.mjs` | 정책 저장, 추천 점수 계산, 설명 생성, 승인 처리 |
| `cli/bin/drctl.mjs` | `drctl policy set`, 추천 조회 CLI |
| `src/App.jsx` | 추천 테이블, 장애 상태, topology, 복구 진행 UI |
| `src/api.js` | 프론트엔드 API endpoint 정의 |

## 개선 가능 지점

현재 구현을 더 엄밀하게 만들려면 다음 개선을 추가할 수 있습니다.

- `rpo` 값을 분 단위로 파싱해 최신 백업 나이와 직접 비교
- Alertmanager `severity`를 점수 산식에 직접 반영
- namespace별 label이나 business priority를 추가 가중치로 반영
- 백업 성공 여부뿐 아니라 restore validation 결과까지 점수화
